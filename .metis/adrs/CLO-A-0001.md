# ADR-1: Trait-based LLM Backend Abstraction

## Context

Clotho uses LLM capabilities in multiple places — speech act extraction from transcripts, entity mention extraction, entity resolution, and potentially future summarization features (e.g., reflection generation, meeting summaries). The initial design assumed Claude API as the sole LLM backend.

However, the LLM landscape is evolving rapidly. Users may want to:
- Use different providers for cost/performance trade-offs (e.g., cheaper models for simple extraction, more capable models for nuanced classification)
- Run local models for privacy-sensitive content
- Swap providers as capabilities and pricing change
- Use different models for different tasks (extraction vs. summarization)

Hardcoding to a single LLM API would create tight coupling and limit flexibility.

## Decision

Define LLM interaction through Rust traits in `clotho-core`, with concrete implementations in `clotho-extract` (and future crates). The backend is selected via configuration in `config/config.toml`.

### Trait Design

```rust
/// Core trait for LLM-powered extraction from transcripts
#[async_trait]
pub trait Extractor: Send + Sync {
    /// Extract speech acts and entity mentions from transcript content
    async fn extract(&self, request: ExtractionRequest) -> Result<ExtractionResult, ExtractorError>;
}

/// Core trait for LLM-powered summarization
#[async_trait]
pub trait Summarizer: Send + Sync {
    /// Generate a summary from content
    async fn summarize(&self, request: SummaryRequest) -> Result<SummaryResult, SummarizerError>;
}

/// Core trait for entity resolution assistance
#[async_trait]
pub trait Resolver: Send + Sync {
    /// Resolve ambiguous entity mentions against known entities
    async fn resolve(&self, request: ResolutionRequest) -> Result<ResolutionResult, ResolverError>;
}
```

### Configuration

```toml
# config/config.toml

[llm]
default_backend = "claude"

[llm.backends.claude]
provider = "anthropic"
model = "claude-sonnet-4-6"
api_key_env = "ANTHROPIC_API_KEY"

[llm.backends.claude-fast]
provider = "anthropic"
model = "claude-haiku-4-5-20251001"
api_key_env = "ANTHROPIC_API_KEY"

[llm.backends.local]
provider = "ollama"
model = "llama3"
endpoint = "http://localhost:11434"

# Task-specific overrides
[llm.tasks]
extraction = "claude"        # Use Sonnet for extraction accuracy
summarization = "claude-fast" # Use Haiku for speed on summaries
resolution = "claude"         # Use Sonnet for resolution accuracy
```

### Implementation Structure

```
clotho-core/src/
    domain/
        traits.rs          # Add Extractor, Summarizer, Resolver traits
        llm_types.rs       # Request/Response types, errors

clotho-extract/src/
    backends/
        mod.rs             # Backend registry, factory
        claude.rs          # Anthropic Claude implementation
        ollama.rs          # Ollama/local model implementation
    pipeline.rs            # Uses dyn Extractor, dyn Summarizer
```

## Alternatives Analysis

| Option | Pros | Cons | Risk Level | Implementation Cost |
|--------|------|------|------------|-------------------|
| Trait abstraction (chosen) | Swappable backends, per-task model selection, testable with mocks | Slightly more upfront design, lowest-common-denominator API | Low | Medium |
| Direct Claude API calls | Simpler initial implementation, can use Claude-specific features | Locked to one provider, hard to test, painful to swap later | Medium | Low |
| Plugin/dynamic loading | Maximum flexibility, third-party backends | Over-engineered for single-user tool, complex error handling | High | High |

## Rationale

The trait approach hits the sweet spot: it's not much more work than direct API calls (one trait + one impl to start), but it gives us clean seams for testing (mock extractors in unit tests) and future flexibility (add backends without touching pipeline code). The configuration layer lets users choose per-task without code changes.

We specifically avoid a plugin/dynamic loading system — that's over-engineering for a single-user tool. If we need a new backend, we add a new impl and recompile.

## Consequences

### Positive
- Can start with Claude, add other backends without pipeline changes
- Per-task model selection (cheap model for summaries, capable model for extraction)
- Easy to mock in tests — no LLM calls needed for unit/integration testing
- Configuration-driven, no code changes to switch models

### Negative
- Trait API must be lowest-common-denominator across providers (can't use Claude-specific features in the trait interface without escape hatches)
- Slightly more initial design work than direct API calls

### Neutral
- Claude remains the default and primary backend — this decision doesn't deprioritize Claude support, it just makes it pluggable
