# ADR-2: Dual-Layer Storage: Relational + Graph with SQLite ATTACH Federation

## Context

The original design used JSONL files (`data/entities.jsonl`, `data/extractions.jsonl`) for structured entity data and graphqlite (`graph/relations.db`) for the relation graph. While JSONL is append-friendly and git-diff-friendly, it has real limitations:

- **No relational queries**: Finding all Tasks belonging to a Program requires scanning the entire file. Joins, filters, aggregations — all require loading everything into memory.
- **No schema enforcement**: JSONL is schema-free, so data integrity relies entirely on application code.
- **Awkward updates**: Modifying an entity in JSONL means rewriting the file or maintaining a log-compaction strategy.
- **Duplicate concerns**: Entity metadata (structured fields) and entity relations (graph edges) are fundamentally different query patterns that benefit from different storage models.

Meanwhile, graphqlite already uses SQLite under the hood. SQLite's `ATTACH DATABASE` allows cross-database queries, meaning a relational entity database and a graph relations database can coexist and even join across boundaries.

## Decision

Adopt a dual-layer storage architecture:

1. **Relational layer** — A SQLite database (`data/entities.db`) for structured entity storage with proper schemas, indexes, and relational queries.
2. **Graph layer** — graphqlite (`graph/relations.db`) for typed relations and Cypher queries, as originally designed.
3. **Federation** — Use SQLite `ATTACH DATABASE` to enable cross-layer queries when needed (e.g., "find all Tasks with state=blocked that are BLOCKED_BY a Blocker mentioning person X").

### Database Layout

```
.workspace/
├── content/              # Markdown (unchanged — human-readable content)
├── data/
│   ├── entities.db       # SQLite — relational entity storage (REPLACES entities.jsonl)
│   ├── extractions.db    # SQLite — draft extractions pending review
│   ├── tags.jsonl        # Keep as JSONL (simple append-only, low query needs)
│   └── events.jsonl      # Keep as JSONL (append-only activity log)
├── graph/
│   └── relations.db      # graphqlite — typed relations (unchanged)
├── index/
│   ├── search.db         # SQLite + FTS5 (unchanged — derived)
│   └── vectors.db        # SQLite + VSS (see CLO-A-0003)
└── config/
    ├── config.toml
    └── ontology.toml
```

### What Stays in JSONL

- `tags.jsonl` — Simple key-value associations, low query complexity, benefits from append-only git diffs
- `events.jsonl` — Pure append-only activity log, never queried relationally, git-diff-friendly

### What Moves to SQLite

- `entities.jsonl` → `entities.db` — Needs relational queries (filter by type, status, date ranges, joins)
- `extractions.jsonl` → `extractions.db` — Needs status tracking, source span lookups, confidence sorting

### Relational Schema (entities.db)

```sql
-- Core entity table (common fields from Entity trait)
CREATE TABLE entities (
    id TEXT PRIMARY KEY,
    entity_type TEXT NOT NULL,
    title TEXT NOT NULL,
    created_at TEXT NOT NULL,
    updated_at TEXT NOT NULL,
    -- Activatable fields (nullable for non-Activatable entities)
    status TEXT,
    -- Taskable fields
    task_state TEXT,
    -- Extractable fields
    extraction_status TEXT,
    source_transcript_id TEXT,
    source_span_start INTEGER,
    source_span_end INTEGER,
    confidence REAL,
    -- ContentBearing fields
    content_path TEXT,
    -- Type-specific JSON for fields that vary by entity type
    metadata JSON
);

CREATE INDEX idx_entities_type ON entities(entity_type);
CREATE INDEX idx_entities_status ON entities(status);
CREATE INDEX idx_entities_task_state ON entities(task_state);
CREATE INDEX idx_entities_extraction_status ON entities(extraction_status);
```

### Federated Query Example

```sql
-- Attach both databases
ATTACH DATABASE 'graph/relations.db' AS graph;
ATTACH DATABASE 'data/entities.db' AS ent;

-- Find all blocked tasks and what's blocking them
SELECT t.title AS task, b.title AS blocker
FROM ent.entities t
JOIN graph.edges e ON e.source_id = t.id AND e.relation_type = 'BLOCKED_BY'
JOIN ent.entities b ON b.id = e.target_id
WHERE t.task_state = 'blocked';
```

### Separation vs. Single Database

The default is **separate databases** (entities.db + relations.db) because:
- Clean separation of concerns (relational vs. graph)
- graphqlite manages its own schema — mixing in application tables risks conflicts
- ATTACH gives federated query when needed
- Can independently backup/restore each layer

If performance profiling shows ATTACH overhead is significant, both can be merged into a single database as an optimization — the code abstracts over the connection, not the file.

## Alternatives Analysis

| Option | Pros | Cons | Risk Level | Implementation Cost |
|--------|------|------|------------|-------------------|
| Dual SQLite + ATTACH (chosen) | Proper relational queries, clean separation, federation when needed, graphqlite untouched | Two DB files to sync, ATTACH has minor overhead | Low | Medium |
| Single SQLite for everything | Simplest deployment, no ATTACH needed | graphqlite schema conflicts, mixing concerns | Medium | Medium |
| Keep JSONL + graph only | Git-friendly diffs, append-only simplicity | No relational queries, poor filter/join performance, awkward updates | High | Low (status quo) |
| Full graph-only (everything in graphqlite) | Single query language (Cypher) | Cypher is awkward for relational queries (status filters, date ranges), fighting the tool | Medium | High |

## Rationale

Entity data is fundamentally relational — you need to filter by type, status, date ranges, and join across entity types. JSONL forces you to load everything into memory for any query. Meanwhile, relations are fundamentally graph-shaped — you need traversals, path queries, and pattern matching. These are different access patterns that benefit from different storage models.

SQLite `ATTACH` gives us the best of both worlds: proper relational queries on entity data, proper Cypher queries on relations, and the ability to join across them when needed. The separation keeps graphqlite's schema clean and lets each layer evolve independently.

## Consequences

### Positive
- Proper relational queries on entity data (indexes, filters, aggregations, joins)
- Schema enforcement at the database level
- Clean updates (UPDATE vs. JSONL rewrite)
- Federated queries across relational + graph layers
- graphqlite schema stays clean and independent

### Negative
- Binary SQLite files have opaque git diffs (mitigated: this was already true for relations.db)
- Two database files to manage in sync
- Slightly more complex initialization (schema migrations)

### Neutral
- Markdown content files unchanged — still human-browsable
- JSONL retained for tags and events where append-only simplicity wins
- Git sync model needs minor update (entities.db replaces entities.jsonl)
