# ADR-4: Visible Content Layout: User-Browsable Filesystem Structure

## Context

Currently all Clotho content lives inside `.clotho/content/` — a hidden directory. This means:

- Users can't browse their notes, meetings, or reflections in a file manager or editor
- Programs, responsibilities, and objectives — the structural skeleton of someone's work — are invisible
- Violates the design principle "content/ is what you browse in any editor"
- Can't open content in Obsidian, VS Code, or any other tool without navigating into hidden dirs
- Git diffs show paths like `.clotho/content/notes/...` which are confusing

Additionally, there is no "inbox" concept — ingestion requires explicit `clotho ingest <file>` calls. For the cadence-driven ceremony model (CLO-I-0010), we need a landing zone where external integrations can continuously deposit materials.

## Decision

Move all human-readable content to **visible directories at the project root**. Keep machine-managed data in `.clotho/`. Add `.clotho/inbox/` as a landing zone for incoming data.

### New Layout

```
my-work/                        # Project root
├── .clotho/                    # Machine-managed (hidden)
│   ├── data/                   # entities.db, extractions.db, tags.jsonl, events.jsonl
│   ├── graph/                  # relations.db
│   ├── index/                  # search.db (gitignored)
│   ├── inbox/                  # Landing zone for incoming data from integrations
│   └── config/                 # config.toml, ontology.toml
│
├── programs/                   # Structural layer — visible
│   ├── technical-education.md
│   └── monolith-breakup.md
├── responsibilities/
│   ├── team-mentorship.md
│   └── budget-management.md
├── objectives/
│   └── reduce-deploy-time.md
├── workstreams/                # Execution layer — visible
│   └── api-redesign.md
├── tasks/
│   ├── write-migration-rfc.md
│   └── implement-api-v2.md
├── meetings/                   # Capture layer — visible
│   ├── 2025-01-15-standup.md
│   └── 2025-01-15-standup.transcript.md
├── reflections/
│   └── 2025-w03-weekly.md
├── artifacts/
│   └── api-design-rfc.md
├── notes/
│   └── architecture-thoughts.md
└── people/                     # Cross-cutting — visible
    └── alice.md
```

### What Moves Where

| Before (.clotho/) | After (project root) |
|---|---|
| .clotho/content/meetings/ | meetings/ |
| .clotho/content/reflections/ | reflections/ |
| .clotho/content/artifacts/ | artifacts/ |
| .clotho/content/notes/ | notes/ |
| .clotho/content/people/ | people/ |
| (not visible) | programs/ |
| (not visible) | responsibilities/ |
| (not visible) | objectives/ |
| (not visible) | workstreams/ |
| (not visible) | tasks/ |

### What Stays Hidden

| Directory | Contents | Why Hidden |
|---|---|---|
| .clotho/data/ | entities.db, extractions.db, tags.jsonl, events.jsonl | Machine-managed databases |
| .clotho/graph/ | relations.db | Binary SQLite, not human-readable |
| .clotho/index/ | search.db | Derived, gitignored |
| .clotho/inbox/ | Incoming files from integrations | Temporary staging area |
| .clotho/config/ | config.toml, ontology.toml | Config is editable but not "content" |

### Inbox Design

`.clotho/inbox/` is a dumb landing zone:
- External integrations push files here continuously (transcription services, calendar sync, Slack export, etc.)
- Files are unprocessed until a ceremony (Daily Debrief) or explicit ingestion processes them
- Processing = ingest into entity system + move content to appropriate visible directory
- What remains in inbox = what hasn't been processed yet
- No schema, no structure — just files with naming conventions

## Alternatives Analysis

| Option | Pros | Cons | Risk Level | Implementation Cost |
|--------|------|------|------------|-------------------|
| Everything in .clotho/ (status quo) | Clean root, simple | Content not browsable, violates design principles | Low | None |
| Content at root, data hidden (chosen) | Browsable in any editor, visible work structure, clean git diffs | More dirs at root, refactor ContentStore paths | Low | Medium |
| Symlinks from root to .clotho/ | Browsable, no data move | Platform-dependent, fragile, confusing | Medium | Low |

## Rationale

The whole point of Clotho's markdown content is that it's human-readable and browsable. Hiding it defeats the purpose. Programs and responsibilities should be visible because they represent the structural skeleton of someone's work — you should be able to see your portfolio by listing a directory.

The inbox enables the cadence-driven ceremony model (CLO-I-0010) where external systems continuously deposit materials and the user processes them at natural time rhythms.

## Consequences

### Positive
- Content browsable in any editor (VS Code, Obsidian, file manager)
- Work structure visible at project root (programs/, tasks/, etc.)
- Clean git diffs on content changes
- Inbox enables integration-driven data flow
- Structural entities (programs, responsibilities) become first-class visible documents

### Negative
- More directories at project root (10 content dirs + .clotho)
- Refactor: ContentStore path resolution, Workspace::init, all tests
- Content paths in entities.db need updating

### Neutral
- .clotho/ still exists for machine data — just smaller
- Git sync behavior unchanged (syncs visible dirs + .clotho/data/ + .clotho/graph/ + .clotho/config/)
