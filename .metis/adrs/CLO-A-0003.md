# ADR-3: Vector Similarity Search (VSS) for Semantic Content Retrieval

## Context

Clotho captures a rich corpus of text: transcript segments, extracted statements (decisions, risks, insights, questions), notes, and reflections. FTS5 handles keyword search well, but many valuable queries are semantic:

- "What have we discussed that's similar to this blocker?"
- "Find past decisions related to this topic"
- "What insights from other programs are relevant to this workstream?"
- "Surface similar risks across different meetings"

These queries require understanding meaning, not just matching keywords. Vector similarity search (VSS) enables this by embedding content into vector space and finding nearest neighbors.

## Decision

Add a SQLite VSS (Vector Similarity Search) layer as a derived index alongside FTS5.

### What Gets Embedded

| Content Type | Granularity | Rationale |
|-------------|-------------|-----------|
| Transcript segments | Per-utterance or per-paragraph | Fine-grained retrieval within meetings |
| Extracted statements | Per-entity (Decision, Risk, Blocker, Question, Insight) | Semantic similarity across extracted facts |
| Notes | Per-section or per-paragraph | Freeform content retrieval |
| Reflections | Per-section | Pattern matching across time periods |
| Artifact summaries | Per-document | Deliverable discovery |

### Storage

```
.workspace/
├── index/                  # Derived cache (gitignored)
│   ├── search.db           # SQLite + FTS5 (keyword search)
│   └── vectors.db          # SQLite + VSS (semantic similarity)
```

`vectors.db` is fully derived — it can be rebuilt from `content/` and `data/` by re-embedding all content. It lives in `index/` (gitignored) alongside `search.db`.

### Schema

```sql
-- Vector embeddings table
CREATE VIRTUAL TABLE vss_embeddings USING vss0(
    embedding(1536)  -- Dimension depends on embedding model
);

-- Metadata linking embeddings to source content
CREATE TABLE embedding_metadata (
    rowid INTEGER PRIMARY KEY,
    entity_id TEXT NOT NULL,
    entity_type TEXT NOT NULL,
    chunk_index INTEGER DEFAULT 0,
    chunk_text TEXT NOT NULL,
    source_path TEXT,
    created_at TEXT NOT NULL
);

CREATE INDEX idx_embed_entity ON embedding_metadata(entity_id);
CREATE INDEX idx_embed_type ON embedding_metadata(entity_type);
```

### Embedding Backend

Embedding generation uses the same trait-based LLM abstraction from CLO-A-0001:

```rust
/// Trait for generating vector embeddings from text
#[async_trait]
pub trait Embedder: Send + Sync {
    /// Generate embeddings for a batch of text chunks
    async fn embed(&self, texts: &[&str]) -> Result<Vec<Vec<f32>>, EmbedderError>;

    /// Embedding dimension for this backend
    fn dimension(&self) -> usize;
}
```

This trait lives in `clotho-core` alongside the other LLM traits. Implementations (e.g., Claude embeddings, OpenAI embeddings, local models) live in `clotho-extract/backends/`.

### Configuration

```toml
# config/config.toml

[llm.backends.embedder]
provider = "voyage"        # or "openai", "local", etc.
model = "voyage-3"
api_key_env = "VOYAGE_API_KEY"

[llm.tasks]
embedding = "embedder"     # Which backend to use for embeddings
```

### Query Flow

```
User query text
    │
    ▼
Embed query via Embedder trait
    │
    ▼
VSS nearest-neighbor search in vectors.db
    │
    ▼
Return ranked results with entity_id + chunk_text
    │
    ▼
Optionally join with entities.db / relations.db via ATTACH
```

### Cross-Layer Queries with ATTACH

VSS results can be combined with relational and graph data:

```sql
-- Find similar content and enrich with entity metadata
ATTACH DATABASE 'data/entities.db' AS ent;
ATTACH DATABASE 'index/vectors.db' AS vec;

SELECT e.title, e.entity_type, vm.chunk_text, vss.distance
FROM vec.vss_embeddings vss
JOIN vec.embedding_metadata vm ON vm.rowid = vss.rowid
JOIN ent.entities e ON e.id = vm.entity_id
WHERE vss_search(vss.embedding, :query_vector)
LIMIT 10;
```

## Alternatives Analysis

| Option | Pros | Cons | Risk Level | Implementation Cost |
|--------|------|------|------------|-------------------|
| SQLite VSS (chosen) | Same SQLite ecosystem, ATTACH federation, portable single file, no external service | Newer extension, smaller community than dedicated vector DBs | Low | Medium |
| Dedicated vector DB (Qdrant, Milvus) | Purpose-built, mature, optimized | External dependency, breaks single-user portable design, overkill for personal tool | Medium | High |
| LLM-powered search only (no vectors) | No index to maintain | Slow (API call per query), expensive, no offline capability | Medium | Low |
| FTS5 only (no semantic search) | Already planned, simple | Misses semantic similarity entirely — keyword matching only | Low | None |

## Rationale

SQLite VSS fits perfectly into Clotho's "everything is SQLite" architecture. It keeps the single-user portable design intact — no external vector database service to run. Being derived (gitignored, rebuildable), it adds no sync complexity. The ATTACH federation from CLO-A-0002 means VSS results can join seamlessly with entity metadata and graph relations.

The `Embedder` trait follows the same pattern as CLO-A-0001, keeping embedding providers pluggable.

## Consequences

### Positive
- Semantic similarity search across all content types
- Stays in the SQLite ecosystem — ATTACH federation with relational and graph layers
- Derived index — no sync burden, rebuildable from source content
- Pluggable embedding backends via trait abstraction

### Negative
- Requires an embedding API (cost per rebuild, though incremental updates mitigate this)
- sqlite-vss is a newer extension — less battle-tested than FTS5
- Adds a fourth trait to the LLM abstraction layer (Embedder alongside Extractor, Summarizer, Resolver)

### Neutral
- Embedding dimension is model-dependent — switching embedding models requires a full re-index
- Chunking strategy (per-utterance vs. per-paragraph) will need tuning
