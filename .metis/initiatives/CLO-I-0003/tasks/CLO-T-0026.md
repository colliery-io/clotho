## Objective

Implement `StoreSync` — the coordination layer that ensures writes are consistent across all backends. Also handles temporal edge materialization (HasCadence/HasDeadline/HasSchedule entity fields → graph edges).

## Acceptance Criteria

## Acceptance Criteria

## Acceptance Criteria

- [ ] `StoreSync` struct holding references to all stores: ContentStore, EntityStore, ExtractionStore, TagStore, EventStore, SearchIndex, GraphStore
- [ ] `save_entity(entity: &dyn Entity + ContentBearing + ...) -> Result<()>` — coordinated write: content file + entities.db + graph node + search index + event log
- [ ] `delete_entity(id: &EntityId) -> Result<()>` — coordinated delete across all backends
- [ ] `materialize_temporal_edges(entity, graph)` — inspects HasCadence/HasDeadline/HasSchedule and creates/updates/removes corresponding graph edges
- [ ] `promote_extraction(id: &EntityId) -> Result<()>` — moves from extractions.db to entities.db, registers graph node, indexes in FTS5, logs event
- [ ] `discard_extraction(id: &EntityId) -> Result<()>` — removes from extractions.db, logs event
- [ ] Event logging on all operations
- [ ] `cargo build` succeeds

## Implementation Notes

### Dependencies
Depends on: CLO-T-0020 through CLO-T-0024 (all individual stores)

### Technical Approach
In `src/sync.rs`. This is the "write coordinator" — all mutations go through StoreSync. Individual stores handle their own backend; StoreSync orchestrates the sequence and ensures consistency. Temporal edge materialization inspects trait implementations to decide which edges to create.
