# Native connectors for external systems (Calendar, Jira, Slack)

*This template includes sections for various types of tasks. Delete sections that don't apply to your specific use case.*

## Feature: Native connectors for external systems

### Priority: P2
### Effort: XL (initiative-level, decompose later)

### Overview

Clotho needs native connectors to sync data from external systems into the workspace. Currently the `.clotho/inbox/` exists as a landing zone but nothing fills it automatically.

### Connector Types

**Capture connectors** (content → entities):
- **Slack** — Thread exports → Transcript entities. Slack conversations are literal transcripts.
- **Google Meet / Otter.ai / transcription services** — Meeting recordings → Transcript entities in inbox
- **Email** — Thread summaries → Note entities

**Reference/pointer connectors** (link, don't copy):
- **Jira** — Tickets → Task entities with URL pointer, sync status/state
- **GitHub** — Issues/PRs → Artifact entities with URL pointer
- **Confluence** — Pages → Artifact entities with URL pointer

**Calendar connectors** (schedule → meetings):
- **Google Calendar / Outlook** — Events → Meeting entities with HAS_SCHEDULE, attendees → Person entities

### Architecture Considerations

- Connectors sync to `.clotho/inbox/` (capture) or directly create entities (reference)
- Each connector needs config in `.clotho/config/` (API keys, sync frequency, filters)
- Prefer vendored/bundled solutions over external service dependencies
- Pointers vs full content: some systems (Jira) should link not copy; others (Slack) should capture
- Idempotency: re-running a sync should not create duplicates (use processing log)
- Incremental: only sync new/changed items, not full history each time

### Future Decomposition

This should become an initiative with per-connector tasks. Likely priority order:
1. Calendar (most immediate value for daily brief/debrief)
2. Slack (transcripts are core to extraction)
3. Jira (task state sync)
4. GitHub (artifact linking)
