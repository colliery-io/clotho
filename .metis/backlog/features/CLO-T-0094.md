# Ontology-driven extraction filtering — prevent capturing irrelevant signals

## Objective

Review and tighten how ontologies are consumed, associated, and defined so that the extraction agents only capture signals the user actually cares about. Currently the debrief-processor extracts too broadly — creating entities for things that aren't relevant to any program or responsibility.

## Problem

- Ontologies define keywords and signal types per program, but the extraction agent doesn't filter tightly enough against them
- Signals get created for topics that don't map to any defined program or responsibility
- No negative signal list ("don't extract this") — only positive matching
- Ontology association is unclear: when should an ontology be consulted? How does the agent decide which program lens to use for a given transcript?

## Priority

- [x] P1 - High (important for user experience)

## Acceptance Criteria

- [ ] Extraction agent only creates entities that match a defined ontology keyword or signal type
- [ ] Unmatched signals are flagged for review rather than auto-created
- [ ] Ontology supports exclusion rules (negative keywords / "ignore" list)
- [ ] Agent clearly documents which ontology it used as the extraction lens for each transcript
- [ ] Reduced noise: measurably fewer irrelevant entities created per debrief

## Implementation Notes

### Areas to investigate
- `plugins/clotho/agents/debrief-processor.md` — how it selects and applies ontologies
- `plugins/clotho/skills/extraction/SKILL.md` — extraction guidance
- `clotho_get_ontology` / `clotho_update_ontology` MCP tools — what fields are available
- Consider adding a confidence threshold: only create entities above a certain relevance score
- Consider a "staging" area where extracted entities are proposed before being committed
