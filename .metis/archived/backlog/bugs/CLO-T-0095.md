# Risk deduplication — detect and merge duplicate risks across extractions

## Objective

The debrief-processor agent creates duplicate Risk entities when the same risk comes up in multiple transcripts/meetings. The same risk appeared 10 times and had to be manually deleted. The agent should detect existing risks before creating new ones, and link additional mentions to the existing entity instead.

## Problem

- Same risk extracted from multiple transcripts creates duplicate entities each time
- No dedup check before creation — agent calls `clotho_create_entity` without searching for existing similar risks
- User had to manually kill 10 copies of the same risk
- The signal that the risk keeps coming up (frequency/recurrence) is lost when duplicates are deleted — that recurrence is actually valuable information

## Priority

- [x] P0 - Critical (blocks users/revenue)

## Desired Behavior

1. Before creating a Risk (or Blocker, Decision, Question), the agent should `clotho_search` for existing entities with similar titles/content
2. If a match is found, create an `extracted_from` relation to the new source transcript instead of creating a duplicate
3. Optionally update the existing entity's content to note recurrence ("Also raised in [transcript] on [date]")
4. The entity's relation count (number of `extracted_from` edges) becomes a signal of how frequently the risk is being raised — this is valuable data, not noise

## Acceptance Criteria

## Acceptance Criteria

- [ ] Debrief-processor searches for existing risks/blockers/decisions/questions before creating new ones
- [ ] Matching entities get a new `extracted_from` relation instead of a duplicate entity
- [ ] Recurrence is noted in the existing entity's content or metadata
- [ ] Frequency signal is preserved — can query "which risks have been raised most often"
- [ ] Zero duplicate risk entities created in a typical daily debrief

## Implementation Notes

### Approach
This is primarily a change to `plugins/clotho/agents/debrief-processor.md` — update the extraction instructions to:
1. Search before create for all derived entity types
2. Define what constitutes a "match" (title similarity, same program context)
3. Add relation + annotation to existing entity when match found

### Abstract over tactical
Risks should be extracted at the **abstract/thematic level**, not the tactical instance level. The agent should recognize that "Spark costs are high" and "S3 costs are growing" from separate conversations are both instances of the same abstract risk: "Infrastructure costs". The abstract risk becomes the entity; the specific tactical mentions become evidence linked via `extracted_from`.

This means dedup isn't just title matching — it's semantic grouping. The agent needs to:
1. Identify the abstract theme behind a tactical signal
2. Search for existing risks at that abstraction level
3. If found, add the new tactical detail as evidence (relation + content annotation)
4. If not found, create the risk at the abstract level with the tactical detail as the first piece of evidence

### Applies to all derived types
While the acute pain is with Risks, the same dedup and abstraction logic should apply to:
- Blockers (same blocker from multiple meetings)
- Decisions (same decision referenced in follow-ups)
- Questions (same open question across sessions)
- Insights (overlapping observations)
